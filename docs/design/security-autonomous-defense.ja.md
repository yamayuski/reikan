# 自律セキュリティ防衛システム — 設計憲章

> **ステータス:** 草案 — Phase 0（アーキテクチャ定義）
>
> 英語版: [security-autonomous-defense.md](security-autonomous-defense.md)
>
> このドキュメントは、Reikan の自律的・AI 主導型セキュリティ防衛システムの憲章とアーキテクチャ意図を定義します。サプライチェーン攻撃・カーネルレベル脆弱性、そして AI ネイティブな脅威環境における先制防衛という包括的な目標を扱います。

---

## 1. 動機

AI ネイティブ時代には、攻撃と防御の双方が加速しています。脅威アクターは AI を活用して以下を実行するようになっています。

- マシン速度で脆弱性を発見・武器化する。
- AI サプライチェーン（モデル重み・学習データ・推論パイプライン）を標的にする。
- パッチが公開される前にカーネルレベルの弱点を悪用する。

Reikan は、人間の速度による脆弱性開示への対応が十分であるという前提を置けません。公開開示から能動的悪用までの時間窓は、数週間から数時間にまで縮小しています。Reikan の設計憲章は、OS がその窓を完全に閉じることを要求します。すなわち、**いかなる人間オペレータより早く**、新たに開示された脆弱性を検知・評価・緩和できなければなりません。

---

## 2. 防衛憲章の核心

> *「攻撃される前に防衛する。Reikan を、攻撃者が標的にしたくないシステムにする。」*

Reikan の自律セキュリティ防衛システムは、以下の交渉不可能な憲章原則に従って運営されます。

| # | 原則 | 説明 |
|---|---|---|
| C-1 | **先制防衛** | システムは既知のエクスプロイトパターンを能動的に追跡し、いかなる攻撃試行が発生するより前に強化を行う。 |
| C-2 | **脆弱性ゼロ前提の否定** | 脆弱性をゼロにすることはできない。目標は、脆弱性ゼロではなく、**露出時間をゼロにする**こと。 |
| C-3 | **人間より速く** | 脆弱性の公開開示から緩和策適用まで、最も優秀な人間のセキュリティチームより速く対応しなければならない。 |
| C-4 | **AI 主導インテリジェンス** | 常時稼働する AI サブシステムが、利用可能なすべてのソースから脆弱性インテリジェンスを継続的に収集する。 |
| C-5 | **防衛深度による抑止** | 防衛層は、Reikan を攻撃することが敵対者にとって合理的でない選択となるほど深くなければならない。 |
| C-6 | **人間の権限の保持** | 不可逆な緩和策や高いシステムリスクを伴う自動緩和策は、実行前に人間の権限者にエスカレーションしなければならない。 |
| C-7 | **監査可能性** | すべての自動セキュリティアクション（検知・評価・緩和・ロールバック）は、システムログ上で完全に監査可能なイベントでなければならない。 |

## 3. 自律脆弱性防衛システム（AVDS）

### 3.1 脅威インテリジェンス収集

AVDS は以下のソースから脆弱性インテリジェンスを継続的に監視・収集します。

- **構造化フィード:** NVD/CVE・OSV・ベンダーセキュリティアドバイザリ（Linux・LLVM・ファームウェア・ハードウェアエラータ）
- **非構造化シグナル:** セキュリティリサーチブログ・SNS（X/Twitter・Mastodon セキュリティコミュニティ）・カンファレンス（Black Hat・DEF CON・USENIX Security）・PoC リポジトリ
- **内部テレメトリ:** 異常なカーネル動作・不審なケイパビリティ使用・予期しないメモリアクセスパターン

収集を担当する AI サブシステム（*脆弱性インテリジェンスエージェント*、VIA）は、各シグナルを以下の軸で分類します。

- **ソース信頼性スコア** — そのソースの過去の精度に基づく重み付け
- **影響コンポーネントスコープ** — Reikan の TCB・カーネル・ランタイム・ユーザー空間コンポーネントのいずれかに影響するか？
- **悪用可能性評価** — PoC は存在するか？野外で能動的に悪用されているか？
- **深刻度** — AI が独自に算出した CVSS 相当スコア（上流スコアとは独立）

### 3.2 脅威の自動トリアージ

信頼性の高いスコープ内シグナルを受信すると、AVDS はトリアージパイプラインに入ります。

```
[シグナル受信]
      │
      ▼
[重複排除・相関分析]  ──→  [既知・対応済み？] ──→ [アーカイブ]
      │
      ▼
[スコープ分析：Reikan の TCB またはカーネルに影響するか？]
      │
  あり │                          なし │
      ▼                               ▼
[深度分析]                     [監視 / 低優先度キュー]
      │
      ▼
[緩和候補生成]
      │
      ▼
[信頼度スコアリング]
      │
   高 │                            低 │
      ▼                               ▼
[自律適用]                    [人間エスカレーションキュー]
      │
      ▼
[検証・ロールバックガード]
      │
      ▼
[監査ログエントリ]
```

### 3.3 緩和策生成とホットパッチ

カーネルレベル・TCB 隣接の脆弱性に対して、AVDS は再起動不要のライブ緩和を目標とします。

- **ケイパビリティ制限:** パッチが適用されるまで、影響を受けるサブシステムのケイパビリティセットを縮小する。
- **IPC/システム呼び出しフィルタリング:** 既知のエクスプロイト呼び出しシーケンスをブロックするポリシールールを注入する。
- **メモリ分離強化:** 影響を受けるコンポーネントのメモリ分離境界を一時的に引き上げる。
- **ホットパッチ適用:** 形式的に検証済みのパッチに対して、再起動なしでカーネルライブパッチを適用する。
- **分離コンテキストでのコンポーネント再起動:** 影響を受ける非重要コンポーネントを、強化された最小権限コンテキストで再起動する。

生成されたすべての緩和策は以下を満たす必要があります。
1. デプロイ前に分離された実行環境でテストされる。
2. 信頼度スコアが割り当てられ、閾値以下のものは人間レビューのために保留される。
3. ポリシーエンジンにロールバックプランを登録した上で適用される。
4. 時間制限あり：設定された期限内に検証できない緩和策はエスカレーションされ、適用されない。

### 3.4 応答時間目標

> **注:** 以下の目標値は**長期的な意図的目標**であり、設計の上限を表すものです。Phase 0 はアーキテクチャ意図を確立するものであり、各フェーズで達成可能なマイルストーンはロードマップ（§33）で定義されます。初期実装では応答時間が大幅に長くなる（分単位ではなく時間単位）可能性があり、AVDS の成熟とともに目標値が引き締められていきます。

| 脆弱性クラス | 長期目標 | Phase 0 基準 |
|---|---|---|
| Reikan カーネルに影響するクリティカル CVE（CVSS ≥ 9.0） | 公開開示から 15 分以内 | 人間支援によるトリアージ |
| システムサービスに影響する高深刻度 CVE（CVSS 7.0–8.9） | 公開開示から 1 時間以内 | 人間支援によるトリアージ |
| サプライチェーン侵害シグナル | 信頼性の高い検知から 30 分以内 | 人間支援によるトリアージ |
| カーネルレベルゼロデイ（CVE なし・PoC あり） | 即時ケイパビリティ制限；4 時間以内にパッチ適用 | 人間支援によるケイパビリティ制限 |

設計意図は、人間の対応チームが主要なボトルネックになってはならないというものです。自動化がこれらの目標を達成するための道であり、開発の各フェーズでこれらの目標に着実に近づいていきます。

### 3.5 人間参加型エスカレーション

以下に該当する緩和策への自動適用は**禁止**されます。

- 同意なしにアクティブなユーザーセッションが依存するケイパビリティやコンポーネントを無効にするもの。
- 形式的チェッカーで検証できない方法でカーネルの TCB を変更するもの。
- 設定された信頼度閾値（デフォルト：0.85 — 誤検知による緩和策を最小化するための保守的な初期値。AVDS の運用データが蓄積されるにつれて経験的に調整が必要）を下回るもの。
- 暗号鍵マテリアルやアテステーションルートに影響するもの。

エスカレーションされた緩和策はオペレータに対して以下とともに提示されます。
- 脆弱性と提案された緩和策の平易な説明文
- AI の信頼度スコアと裏付け証拠
- ロールバック手順
- 脆弱性が能動的に悪用されると推定されるまでのカウントダウンタイマー

### 3.6 ロールバックとリカバリ

AVDS が適用するすべての緩和策は、適用前にポリシーエンジンに対応するロールバック手順を登録しなければなりません。ロールバック手順は以下を満たさなければなりません。
- 緩和策がシステム不安定を引き起こした場合、人間の介入なしに実行可能であること。
- 緩和策のデプロイ前に分離環境でテストされていること。
- インシデント後の分析のために監査ログに保持されること。

## 4. サプライチェーンセキュリティ

### 4.1 ビルド時の整合性

ソースコード・コンパイラツールチェーン・ファームウェア・ハードウェアマイクロコード・AI モデル重みを含む、すべてのビルド入力物は暗号的プロベナンス検証の対象となります。

- すべてのアーティファクトはコンテンツアドレス型ハッシュで識別される。
- ビルドグラフ（入力 → 出力）は保存され、再現可能である。
- 信頼済みルートまで追跡可能な検証済みプロベナンスチェーンなしに、いかなるアーティファクトもビルドパイプラインに入ることはできない。

ビルド環境自体は密閉型で封印されます。
- コンパイル中はネットワークアクセスなし。
- すべてのビルド依存関係はピン留めされ、密閉キャッシュに事前取得される。
- ビルド環境は各ビルド前に既知の正常なアテステーションと照合して検証される。

### 4.2 依存関係リスク監視

AI 支援による*依存関係リスクエージェント*（DRA）が、すべての直接・推移的依存関係を継続的に監視します。

- **メンテナアカウント侵害**シグナル（異常なコミットパターン・新しいメンテナ・行動上の異常）
- **タイポスクワッティング** — 正規の依存関係に類似した名前のパッケージ
- **隠れたバックドア** — 疑わしいパターン（データ窃取・遅延実行・環境プロービング）に対する静的・意味論的コード解析
- **ライセンスドリフト** — プロジェクトポリシーに違反する依存関係ライセンスの変更
- **バージョンピン留め違反** — 依存関係グラフにおける予期しないバージョンバンプ

高信頼度のサプライチェーンリスクシグナルを検知した場合、DRA は以下を行います。
1. 影響を受ける依存関係を最後の既知の正常なバージョンで凍結する。
2. 代替依存関係の提案を生成する（利用可能な場合）。
3. 完全な証拠パッケージとともに人間レビューにエスカレーションする。

### 4.3 ランタイムアーティファクト検証

ロード時に、カーネルモジュール・システムサービスバイナリ・AI モデル重みを含むすべての実行アーティファクトは、登録済みプロベナンスエントリと照合して検証されます。検証に失敗したアーティファクトは以下の対応を受けます。
- 拒否・隔離される。
- 監査ログにセキュリティイベントとして記録される。
- 自動インシデント対応（AVDS パイプライン）の対象となる。

## 5. カーネルレベル先制防衛

### 5.1 継続的カーネル完全性監視

カーネルはライブ不変条件チェックを実行し、以下を継続的に検証します。
- ケイパビリティテーブル完全性（不正なケイパビリティエントリなし）
- メモリ領域オーナーシップ（2 つの分離コンテキストが同一領域を同時所有しない）
- スケジューラポリシー準拠（無制限リソースを消費する暴走エージェントなし）
- IPC チャネル有効性（すべてのオープンチャネルが一致するケイパビリティを持つ有効なエンドポイントを持つ）

不変条件違反は即時自動対応をトリガーします。
- 違反コンテキストの終了
- フォレンジック分析のためのシステム状態スナップショット
- AVDS パイプラインへのアラート

### 5.2 エクスプロイト技法フィンガープリンティング

カーネルは既知のエクスプロイト技法シグネチャ（ROP チェーンパターン・ヒープスプレーフィンガープリント・ケイパビリティ混同パターン）のライブラリを維持します。実行モニターはこれらのパターンをパッシブにチェックします。

- 実行中エージェントの命令ストリーム
- メモリ割り当てパターン
- IPC メッセージシーケンス

検知は即時コンテキスト停止と AVDS エスカレーションをトリガーします。

### 5.3 投機的実行とサイドチャネル分離

Reikan のカーネルは厳格な投機的実行境界を強制します。
- コンテキストスイッチ境界で、セキュリティドメインをまたぐ投機的実行を無効化する。
- TLB・BTB・L1/L2 キャッシュなどの共有マイクロアーキテクチャ状態は、感度差がパフォーマンスコストを正当化するドメイン遷移時にフラッシュされる。
- GPU コンピュートコンテキストは、ハードウェアスケジューラレベルで CPU コンテキストから分離される。
- 非特権コンテキストに公開されるタイミング観測可能な操作に制御されたノイズを追加することで、タイミングチャネルを軽減する。

### 5.4 カーネル自己アテステーション

設定可能な間隔で、および重要なセキュリティイベント発生時に、カーネルは自身の完全性について署名付きアテステーションを生成します。
- アテステーションの対象：ロード済みモジュール・ケイパビリティテーブルハッシュ・アクティブポリシールール・メモリレイアウト
- アテステーションはシステムの公開アテステーション鍵を使用して外部関係者が検証可能
- アテステーション失敗は AVDS クリティカル対応パイプラインをトリガー

## 7. 未解決の問題

- **Q1:** VIA が応答時間目標を確実に達成するために必要な AI モデルの最小サイズ・最小能力は何か？
- **Q2:** AVDS が競合するシグナルを処理する場合（例：Reikan が形式的に検証済みのサブシステムに影響する CVE）はどうすべきか？形式的証明は十分な緩和策となるか？
- **Q3:** アテステーション鍵階層を管理する暗号方式は何か？またダウンタイムなしで鍵をローテーションするにはどうするか？
- **Q4:** 誤って生成されたパッチを適用するリスクを考慮した場合、カーネルライブパッチの自律適用に対する閾値ポリシーをどうすべきか？
- **Q5:** セキュリティ判断に使用する自身の AI モデルのサプライチェーン侵害を検知した場合、システムはどのように振る舞うべきか？
