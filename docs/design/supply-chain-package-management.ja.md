# サプライチェーンパッケージ管理 — 設計

> **ステータス:** 草案 — Phase 0（アーキテクチャ定義）
>
> 英語版: [supply-chain-package-management.md](supply-chain-package-management.md)
>
> このドキュメントは、`reikan add` の組み込み防衛的依存関係管理ワークフロー（ソース同一性固定・副作用宣言/承認・推移依存のポリシー評価）を定義します。

---

## 1. スコープと目標

本設計では、以下のように外部ライブラリを追加できる組み込みサプライチェーン管理を導入します。

```bash
reikan add https://github.com/<owner>/<repo>.git
```

システムは以下を満たします。

1. コードに `eval` や意図的な難読化/マングルが含まれるライブラリを自動拒否する。
2. ライブラリの副作用をデフォルトで全面禁止する。
3. ライブラリメタデータ JSON で副作用宣言を必須にする。
4. `reikan add` 時に副作用要求に対するユーザー同意を取る。
5. すべての推移依存に対して要求を連鎖評価する（依存関係全体の要求を満たす必要がある）。
6. メタデータの commit ID によるソース一意性固定を必須にする。
7. `dependencies` / `devDependencies` のユーザー区分を不要にし、ビルド時 AST tree-shaking で不要依存を除外する。

---

## 2. 信頼モデルとソース同一性

各ライブラリの発行物メタデータには以下を含めます。

- 正準ソースリポジトリ URL
- 不変なソース commit ID（または同等の content-addressed 識別子）
- 宣言された副作用要求

commit ID は再現性とポリシー評価の信頼根拠です。ソースとメタデータの同一性が一致しない場合、依存追加は拒否します。

---

## 3. メタデータ契約（草案）

```json
{
  "name": "sample-lib",
  "version": "0.1.0",
  "source": {
    "repo": "https://github.com/yamayuski/sample-lib.git",
    "commit": "3f2e7b8c..."
  },
  "sideEffects": [
    {
      "kind": "network",
      "reason": "Fetches remote model metadata at runtime"
    }
  ],
  "dependencies": [
    {
      "repo": "https://github.com/example/child-lib.git",
      "commit": "7af9e1d2..."
    }
  ]
}
```

補足:

- `sideEffects` は必須。空配列は「副作用なし」を意味する。
- 未宣言副作用はポリシー違反とみなす。
- 依存エントリは不変 commit に固定する。

---

## 4. `reikan add` セキュリティフロー

1. パッケージソースとメタデータを解決する。
2. メタデータとソースの同一性固定（`repo` + `commit`）を検証する。
3. パッケージソースに静的ガードレールスキャンを実行する。
   - `eval` を検知したら拒否する。
   - 強い難読化/マングルパターンを検知したら拒否する。
4. 推移依存を再帰的に解決し、同じ検証を繰り返す。
5. 依存グラフ全体から副作用要求を集約する。
6. ユーザーに明示的確認を求める。
   - 「このライブラリ（および依存）が以下の副作用を要求しています。許可しますか？」
7. 要求副作用をすべて承認した場合のみ依存へ追加する。
8. 再現可能ビルドのため lock と承認ポリシーを保存する。

依存チェーン上のどこか 1 つでもスキャン/同一性/ポリシー検証に失敗した場合、追加処理全体をアトミックに失敗させます。

---

## 5. 副作用ポリシー意味論

- デフォルトポリシーは「副作用 deny-all」。
- 副作用は明示的なユーザー承認でのみ許可される。
- 承認評価は直接依存だけでなく依存閉包全体に対して行う。
- 実行時挙動は承認済み宣言を超えてはならない。

---

## 6. 依存分類ポリシー

npm などと異なり、Reikan は利用者に runtime 依存（`dependencies`）と debug/build-time 依存（`devDependencies`）の分類を要求しません。

Reikan のビルドは AST ベース tree-shaking により未使用依存を成果物から除外します。そのため依存宣言は単一トラックで扱い、成果物への同梱可否はビルド時の意味的使用実態に基づいて決定します。

---

## 7. 目標セキュリティ性質

- **サイレント副作用なし:** すべての副作用は宣言され、同意される。
- **ピン留めなしのソースドリフトなし:** commit 単位の同一性固定を必須化する。
- **不審コードの取り込みなし:** `eval`/難読化ブロックを add 時に強制する。
- **部分チェーン信頼バイパスなし:** 推移依存要求を常に評価する。

---

## 8. 未解決設計論点

1. 難読化検知の閾値設計と誤検知時の取り扱い。
2. 副作用分類体系（network/file/process/FFI など）の標準化。
3. 承認スコープ粒度（プロジェクト単位・依存単位・バージョン単位・commit 単位）。
4. 推移依存の副作用宣言がバージョン更新で変化した場合のポリシー更新フロー。
