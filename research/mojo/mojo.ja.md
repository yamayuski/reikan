# Mojo

> English version: [mojo.md](mojo.md)

---

## 概要

**Mojo** は **Modular**（LLVM と Swift の作者 Chris Lattner が設立）によって設計され、2023 年に発表されたプログラミング言語です。目標は AI のためのシステム言語であること：高性能なネイティブコードにコンパイルでき、**MLIR** を通じて異種ハードウェア（CPU、GPU、TPU、アクセラレータ）をターゲットにでき、AI 研究（Python エルゴノミクス）と AI システムプログラミング（C ライクなパフォーマンス）の両方のための単一言語として機能する Python のスーパーセットです。

Mojo は値セマンティクスを持つ `struct` 型、Rust にインスパイアされた所有権モデル、SIMD 組み込み命令、そして MLIR を通じてハードウェア固有バックエンドへの段階的な低下を導入しています。AI インフラ（CUDA カーネル、カスタムオペレーション、ランタイム内部）で C/C++ を最終的に置き換え、Python エコシステムとの相互運用性を維持するように設計されています。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| パラダイム | マルチパラダイム：命令型、OO、関数型要素 |
| Python 互換性 | Python のスーパーセット — 有効な Python は有効な Mojo（一部注意点あり） |
| 型システム | 静的型 + 動的 Python 型；値セマンティクスを持つ `struct`；所有権モデル |
| コンパイル | MLIR → LLVM IR → ネイティブコードを通じた AOT および JIT |
| ハードウェアターゲット | CPU（SIMD）、GPU（MLIR GPU ダイアレクト経由）、アクセラレータ |
| メモリモデル | 所有権 + 借用（Rust にインスパイア）；SIMD 対応レイアウト |
| MLIR 統合 | 第一級；Mojo は MLIR の上に構築 |
| 相互運用性 | Python 相互運用；C/C++ FFI |
| 形式検証 | なし |
| AI フォーカス | 中心的 — AI/ML ワークロードとインフラのために設計 |
| 開発元 | Modular Inc.（プロプライエタリ + 部分的 OSS） |

### MLIR コンパイルパイプライン

```
Mojo ソース
     ↓
Mojo IR（高レベル）
     ↓
MLIR ダイアレクト（affine、gpu、vector、tensor など）
     ↓
LLVM IR
     ↓
ネイティブ機械語（CPU / GPU / アクセラレータ）
```

## Reikan との類似点

| 観点 | Mojo | Reikan |
|---|---|---|
| **設計目標** | AI ワークロード向けに作られた言語 | 専用言語（Reikan-lang）を持つ AI-First OS |
| **AI を主要な関心事とする** | AI/ML インフラが言語存在の理由 | AI エージェントと推論が第一級の OS 構成要素 |
| **異種計算** | MLIR 経由で CPU・GPU・アクセラレータをネイティブにターゲット | GPU・NPU・VRAM・アクセラレータが第一級のスケジュール対象リソース |
| **所有権モデル** | GC なしのメモリ安全性のための Rust インスパイア所有権 | Reikan-lang はメモリ安全性と形式検証可能性をターゲット |
| **MLIR ベースコンパイル** | Mojo はハードウェア移植性のために MLIR を通じてコンパイル | Reikan のコンパイラ/IR は MLIR または同等の段階的低下パイプラインをターゲットにすべき |
| **一言語でシステムと AI** | AI 研究エルゴノミクス + システムプログラミングパフォーマンス | Reikan-lang はカーネルレベルのシステムコードとエージェントレベルのロジックの両方のために設計 |
| **レガシーからの決別** | Python スーパーセットだが、システム層は C/C++ の前提を放棄 | Reikan は POSIX・ELF・C/C++ の前提を放棄 |
| **パフォーマンスフォーカス** | 明示的な最適化ターゲットとしての推論レイテンシとスループット | 推論レイテンシとトークンスループットが主要な非機能要件 |

## Reikan との相違点

| 観点 | Mojo | Reikan |
|---|---|---|
| **スコープ** | プログラミング言語（ユーザー空間と ML インフラ） | OS + システム言語 |
| **OS プリミティブ** | なし — Mojo は言語であり OS ではない | カーネル、スケジューラ、ケイパビリティマネージャ、エージェントランタイム |
| **Python スーパーセット** | 明示的な Python スーパーセットのコミットメント | 言語レベルでの Python 互換性レイヤーなし |
| **形式検証** | なし — 証明ベースの正当性なし | TCB の機械検証形式証明 |
| **ケイパビリティセキュリティ** | ケイパビリティモデルなし | 完全なケイパビリティベースセキュリティ |
| **エージェント抽象化** | エージェント OS の概念なし | 目標・メモリ・ツール・ライフタイムを持つエージェントがネイティブ OS オブジェクト |
| **可観測性** | 標準的なプロファイリングとトレーシングツール | ケイパビリティ付与とモデル呼び出しがすべて均一な監査可能イベント |
| **所有権** | プロプライエタリ言語（Modular）、部分的 OSS | Reikan は完全オープン |
| **LLM 記述** | LLM コード生成向けに特別設計されていない | Reikan-lang は人間と LLM の両方によって記述されるように設計 |
| **ハードウェア隔離** | ユーザー空間言語；隔離は OS に依存 | OS のハードウェア強制ケイパビリティ隔離 |

## 参考にすべき点

### 1. コンパイルターゲットとしての MLIR
Mojo のアーキテクチャは、MLIR が単一のソース言語から CPU・GPU・特化アクセラレータをターゲットにする必要がある言語のための実用的な汎用コンパイル基板であることを実証しています。Reikan のコンパイラ（Reikan-lang のための）は、異種低下パイプラインを再発明するのではなく、中間表現として MLIR を真剣に評価すべきです。

### 2. 段階的低下
Mojo は MLIR の段階的低下モデルを使用します：高レベルのセマンティック表現が段階的にハードウェア固有の操作に低下します。これにより、高レベルの AI コードが移植性を失わずに CUDA カーネルや Apple Neural Engine のオペレーションにマップされます。Reikan-lang は異種計算抽象化をサポートするために同じ段階的低下の哲学を採用すべきです。

### 3. AI データのための値セマンティクス
Mojo の `struct` 型はデフォルトで値セマンティクスを持ちます — Python の参照セマンティクスからの大きな逸脱です。これによりテンソル型の効率的なスタック割り当てと SIMD 対応データレイアウトが可能になります。Reikan-lang の型システムは、パフォーマンスクリティカルなエージェント状態とモデルアクティベーション型に対して同様に値セマンティクスを優先すべきです。

### 4. 借用チェッカーの摩擦のない所有権
Mojo の所有権モデルは、コールサイトでの摩擦（明示的なライフタイムアノテーションの削減）を少なくして Rust の安全性保証を提供することを目指しています。Reikan-lang はさらに進む機会があります：形式検証が設計目標であるため、所有権モデルはエルゴノミクスだけでなく、証明しやすさを第一級の関心事として設計できます。

### 5. 研究とシステムのための一言語
Mojo の賭け — 単一の言語が AI 研究（Python エルゴノミクス）とシステムプログラミング（C パフォーマンス）の両方に対応できる — は、Reikan-lang をエージェントレベルのロジックとカーネルレベルのシステムコードの両方に対応するように設計するという Reikan のアプローチを検証します。教訓：型システムと所有権モデルは、異なる方言なしに高レベルの抽象化と低レベルの制御の両方をサポートするように設計される必要があります。

### 6. 第一級言語の関心事としての推論レイテンシ
Mojo は推論スループットとレイテンシを明示的な設計軸にしています：SIMD 組み込み命令、キャッシュフレンドリーなデータレイアウト、GPU カーネルディスパッチが第一級の言語機能です。Reikan は Reikan-lang の型システムに同様のパフォーマンス契約をエンコードすべきです — 特に VRAM 帯域幅と推論レイテンシが支配的な制約であるモデルランタイム層において。
