# AIOS — LLM エージェント・オペレーティングシステム

> English version: [aios.md](aios.md)

---

## 概要

**AIOS**（LLM Agent Operating System）は、ラトガーズ大学（Mei、Xu ら、2024）による研究プロジェクトおよびプロトタイプです。複数の並行 LLM エージェントをホストするために特別に設計された OS レベルの抽象化層を提案しており、LLM エージェントが従来のアプリケーションとは根本的に異なるリソース要件を持つという課題に取り組んでいます：モデルウェイトのために大量の VRAM を消費し、長時間実行のコンテキストウィンドウを必要とし、外部ツールを呼び出し、推論計算と I/O の両方のきめ細かなスケジューリングを必要とします。

AIOS は既存の OS（Linux）の**上に**構築され、ホスト OS を置き換えるのではなく、エージェント中心のカーネルモジュール層を追加します。この階層化アプローチにもかかわらず、その設計空間 — エージェントスケジューリング、エージェントのメモリ管理、ツール呼び出し管理、アクセス制御 — は Reikan の AI-First OS の使命と直接重複しています。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| アーキテクチャ | Linux の上のエージェント OS 層 |
| 主要言語 | Python |
| エージェント抽象化 | 目標・コンテキスト・ツールアクセス・ライフサイクルを持つ LLM エージェント |
| スケジューリング | エージェントスケジューラ：複数 LLM エージェントの実行をインターリーブ |
| メモリ管理 | コンテキストウィンドウ管理；モデルウェイトキャッシュ |
| ツール管理 | ツールレジストリ；ツール呼び出しディスパッチ；サンドボックス化 |
| アクセス制御 | エージェントごとのツールおよびリソース権限 |
| ストレージ | エージェント状態の永続化；知識検索 |
| 形式検証 | なし |
| AI フォーカス | 中心的 — エージェントと LLM 推論が主要な構成要素 |

### AIOS のコアコンポーネント

| コンポーネント | 機能 |
|---|---|
| **エージェントスケジューラ** | 複数 LLM エージェントの実行をインターリーブ；長い LLM 呼び出し中の待機状態を処理 |
| **コンテキストマネージャ** | エージェントコンテキストウィンドウを管理；コンテキストの永続化と復元を処理 |
| **メモリマネージャ** | エージェントの短期（作業）記憶と長期（ストレージ）記憶を管理 |
| **ツールマネージャ** | 外部ツール呼び出しを登録・ディスパッチ；ツール権限を管理 |
| **アクセスマネージャ** | ツールとストレージに対するエージェントごとの権限モデル |
| **LLM システムコール** | エージェントと AIOS カーネル間の安定した API を定義 |

## Reikan との類似点

| 観点 | AIOS | Reikan |
|---|---|---|
| **主要設計軸** | LLM エージェントを計算の主要単位とする | AI エージェントを OS の第一級市民とする |
| **エージェントライフサイクル** | 目標・コンテキスト・メモリ・ツールアクセスがシステムによって管理されるエージェント | 目標・メモリ・ツール・ライフタイムを持つエージェントがネイティブカーネルオブジェクト |
| **エージェントスケジューラ** | LLM エージェント専用スケジューラ | AI 推論を第一級ワークロードとする異種スケジューラ |
| **メモリ抽象化** | エージェント固有のメモリ管理（コンテキスト、KV キャッシュ、永続メモリ） | エージェントとモデルのワークロードを中心に設計されたメモリオブジェクトモデル |
| **ツール呼び出し** | 権限制御を伴うマネージドツールディスパッチ | ケイパビリティ仲介によるツールおよびリソースアクセス |
| **アクセス制御** | エージェントごとの権限モデル | ケイパビリティベースセキュリティ — 権限は明示的かつエージェントごと |
| **LLM 固有スケジューリング** | 長い推論呼び出し・待機状態・コンテキストスイッチングを処理 | 異種リソースの推論対応スケジューリング |
| **エージェント API** | LLM システムコール API | エージェント OS API（コンテキスト・ツール呼び出し・セマンティッククエリ・アテステーション） |

## Reikan との相違点

| 観点 | AIOS | Reikan |
|---|---|---|
| **アーキテクチャの深さ** | Linux の上の層 — 新しい OS ではない | ゼロからの OS 設計；ホスト OS 基板なし |
| **カーネルセキュリティ** | Linux のセキュリティモデルを継承；ケイパビリティベースカーネルなし | 形式的セキュリティ特性を持つ完全なケイパビリティベースカーネル |
| **形式検証** | なし | TCB の機械検証証明 |
| **異種計算** | Linux GPU ドライバーに依存；ネイティブアクセラレータスケジューリングなし | GPU・NPU・VRAM がカーネルの第一級スケジュール対象リソース |
| **言語** | Python — 高レベル；システム言語ではない | Reikan-lang — カーネルとエージェントのためのシステムレベル言語 |
| **可観測性** | 標準ログ；均一な監査ではない | ケイパビリティ付与とモデル呼び出しがすべて監査可能なイベント |
| **ケイパビリティセキュリティ** | Linux から継承した POSIX 権限 | 偽造不能なケイパビリティトークン；アンビエント権限なし |
| **ハードウェア隔離** | Linux によるプロセス/コンテナ隔離 | ハードウェア強制ケイパビリティ隔離 |
| **スコープ** | 研究プロトタイプ | プロダクション向けアーキテクチャ |
| **ブートモデル** | 既存の Linux インストール上で動作 | ベアメタルブートチェーン |

## 参考にすべき点

### 1. エージェント OS API 設計
AIOS の「LLM システムコール」インターフェースは、LLM エージェントと OS 層の間の安定した API を定義する具体的な試みです。定義された動詞 — モデル推論、コンテキスト保存/復元、ツールディスパッチ、メモリクエリ — は、Reikan のエージェント OS API 設計（アーキテクチャアウトラインのセクション 21）の有用な最初の草案です。Reikan は AIOS の呼び出しのどれが直接翻訳され、どれがケイパビリティトークンを中心に再設計する必要があるかを評価すべきです。

### 2. エージェントスケジューリングのセマンティクス
AIOS のエージェントスケジューラは、LLM ワークロード固有の課題を処理します：推論呼び出しは長時間実行され、トークンレベルでは中断できません。スケジューラはこれらの長い待機状態（LLM 応答を待つ、ツール応答を待つ）の周りでエージェントの実行をインターリーブします。これは Reikan の異種スケジューラへの直接の仕様入力です：マイクロ秒粒度のハードウェアスケジューリングと秒粒度のエージェント実行インターリーブの両方を処理する必要があります。

### 3. 管理リソースとしてのコンテキストウィンドウ
AIOS は LLM **コンテキストウィンドウ**を単なるヒープ割り当てではなく、希少な管理リソースとして扱います。コンテキストウィンドウは複数のエージェント間で保存・復元・管理されます。これは Reikan の「コンテキスト」を第一級カーネルオブジェクトとすることを動機付けます：カーネルは VRAM と DRAM にわたるコンテキストウィンドウ割り当てを追跡し、コンテキスト移行を処理し、エージェントごとのコンテキスト制限を強制する必要があります。

### 4. ツール権限モデル
AIOS のエージェントごとのツール権限モデル（どのエージェントがどのツールをどの制限で呼び出せるか）は、Reikan のケイパビリティベースツールアクセスモデルの直接の先例です。Reikan の重要な改善点：フラットな権限リストの代わりに、ツールアクセスは偽造不能なケイパビリティトークンによって仲介され、監査可能かつ形式的に合成可能になります。

### 5. エージェントメモリの分類
AIOS はエージェントのメモリを短期（コンテキスト内）、作業（一時的外部）、長期（永続的）に区別します。この分類は Reikan のメモリオブジェクトモデルの有用な出発点で、VRAM（ホットモデルアクティベーション）、DRAM（エージェント作業状態）、ストレージ（永続的知識）を異なるスケジューリングと移行特性を持つ異なるリソースクラスとして処理する必要があります。
