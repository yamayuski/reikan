# Redox OS

> English version: [redox-os.md](redox-os.md)

---

## 概要

**Redox OS** は、Jeremy Sousa と貢献者らによって 2015 年から開発されているオープンソースの Unix 系 OS で、全体が **Rust** で書かれています。Linux、GNU、または既存の Unix のフォークではなく、マイクロカーネルをベースにゼロから設計された OS です。マイクロカーネル構造を持ちながら、Bash、GCC、SDL2 などの既存ソフトウェアを実行するための POSIX 互換性をある程度提供しています。

Redox の中心的な主張は、Rust の所有権モデルと型システムが安全な OS を構築するための正しいツールであるというものです：形式証明によってではなく、言語レベルで強制されるメモリ安全性です。**URL スキームベースのリソースモデル**も注目すべき特徴で、ファイル、ネットワークソケット、ハードウェアデバイスなど、ほぼすべてのものがユーザー空間の「スキームプロバイダー」によって処理される URL としてアドレス指定されます。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| カーネル | Redox マイクロカーネル — 最小限、Rust で書かれている |
| 主要言語 | Rust |
| IPC | ファイルディスクリプタ的なスキームインターフェース |
| リソースモデル | URL スキーム（`file:`、`tcp:`、`display:`、`orbital:` など） |
| セキュリティモデル | スキームベースの隔離；名前空間；部分的ケイパビリティモデル |
| POSIX 互換性 | 部分的（Redox 独自の C 標準ライブラリ relibc 経由） |
| 形式検証 | なし — Rust の型安全性に依存 |
| AI フォーカス | なし — 汎用デスクトップ/組み込み OS |
| GUI | Orbital — 最小限の Wayland インスパイア コンポジタ |

## Reikan との類似点

| 観点 | Redox OS | Reikan |
|---|---|---|
| **出自** | Linux や Unix のフォークではない | Linux、Windows、macOS のフォークではない |
| **アーキテクチャ** | マイクロカーネル — 最小限のカーネル、サービスはユーザー空間 | 検証済みマイクロカーネル + システムサービス層 |
| **主要言語** | Rust — メモリ安全性をコア特性として | Reikan-lang；ブートストラップに Rust を使用 |
| **安全性の哲学** | 言語レベルで強制されるメモリ安全性 | 言語と形式証明の両方で強制される安全性 |
| **スキームモデル** | URL ベースのリソースアドレス指定（汎用インターフェース） | ケイパビリティ仲介アクセスを持つオブジェクトストア |
| **マイクロカーネルの規律** | ドライバーとサービスはユーザー空間 | 検証済みカーネルの上にドライバーとサービス |
| **Rust エコシステム** | Rust エコシステムの完全統合 | システムサービス層に Rust を使用可能 |

## Reikan との相違点

| 観点 | Redox OS | Reikan |
|---|---|---|
| **POSIX 互換性** | relibc による部分的 POSIX — 設計上の妥協 | 設計上 POSIX なし；互換性は将来のオプトインレイヤー |
| **AI フォーカス** | 汎用デスクトップ OS；AI 固有の抽象化なし | AI エージェント・モデル推論・LLM パイプラインが第一級の OS 構成要素 |
| **形式検証** | なし — Rust の型チェッカーのみに依存 | TCB の機械検証形式証明が設計目標 |
| **ケイパビリティモデル** | スキーム名前空間；完全なケイパビリティセキュリティではない | 偽造不能な権限トークンによる完全なケイパビリティベースセキュリティ |
| **異種計算** | GPU/NPU スケジューリングなし | GPU・NPU・VRAM・アクセラレータが第一級のスケジュール対象リソース |
| **エージェント抽象化** | エージェント概念なし | 目標・メモリ・ツール・ライフタイムを持つエージェントがネイティブカーネルオブジェクト |
| **可観測性** | 標準的な Unix 型ログ | ケイパビリティ付与とモデル呼び出しがすべて監査可能なイベント |
| **開発モデル** | 従来型のオープンソースコミュニティ | AI 主導開発（LLM がアクティブなコントリビューター） |
| **GUI スタック** | Orbital コンポジタ | AI ネイティブシェル、ワークスペースグラフ、会話型コントロール |

## 参考にすべき点

### 1. OS 実装のための Rust
Redox は Rust が OS カーネルとシステムサービスのエンドツーエンドで使用できることを示す最も完成した実証事例です。実際に遭遇した課題 — カーネルコード内の `unsafe` ブロック、FFI 境界、ハードウェア固有の組み込み命令、割り込みハンドラ — は、ブートストラップ層とシステムサービスに Rust を使用する Reikan に直接関連します。

### 2. URL スキームリソースモデル
Redox のスキームモデルは Unix の「すべてはファイル」哲学のクリーンな汎化です：ファイルシステム階層の代わりに、リソースは URL スキームでアドレス指定され、任意のユーザー空間サービスがスキームプロバイダーとして登録できます。これは POSIX ファイルディスクリプタよりもシンプルで構成可能なインターフェースです。Reikan はエージェント/オブジェクトストアのインターフェース層で類似の抽象化を探求できます。

### 3. マイクロカーネルサービス隔離
Redox は、ネットワークドライバー、ディスプレイドライバー、ストレージドライバーがすべてマイクロカーネルのユーザー空間に存在できることを、禁止的なパフォーマンスオーバーヘッドなしに実証しています。これは Reikan のマイクロカーネルアーキテクチャが理論的に望ましいだけでなく実用的であることの証拠です。

### 4. `relibc` — 互換性レイヤー設計からの教訓
クリーンルーム C 標準ライブラリ（relibc）の開発によって POSIX 互換性をカーネル変更なしに実現した Redox の事例は有用な教訓です：互換性レイヤーはユーザー空間で完全に実装でき、カーネルをレガシーの制約から自由に保てます。Reikan は POSIX 互換性が必要になった場合、カーネルの妥協なしにユーザー空間レイヤーとして追加できることを記録しておくべきです。

### 5. コミュニティ主導の Rust OS 開発
10 年にわたるオープンソース・コミュニティ主導プロジェクトとしての Redox の開発史は、Reikan のコントリビューションモデルの有用な参照事例です — 特に、多くの貢献者との長期にわたる複雑な Rust コードベースの管理において。
