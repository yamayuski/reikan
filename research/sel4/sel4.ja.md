# seL4 マイクロカーネル

> English version: [sel4.md](sel4.md)

---

## 概要

**seL4** は NICTA（現 CSIRO の Data61）と seL4 Foundation が開発した高保証マイクロカーネルです。**機械検査による関数正当性の形式証明**を受けた最初の OS カーネルであり、カーネルの C 実装が高レベルの抽象仕様と証明可能に一致しています。正当性に加えて、セキュリティ強制（完全性と機密性）の証明、および一部構成ではリアルタイム認定のための最悪実行時間境界も備えています。

seL4 は安全性・セキュリティが重要なシステム向けに設計されており、航空電子、自動車、産業制御、防衛分野で展開されています。OS セキュリティ研究における「形式検証済み」の意味する参照点です。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| カーネル | 第三世代 L4 マイクロカーネル |
| 主要言語 | C（カーネル）；Rust ポート開発中（ferrocene、microkit） |
| 証明支援系 | Isabelle/HOL |
| 証明内容 | 関数正当性、完全性、機密性、WCET（部分的） |
| セキュリティモデル | ケイパビリティベース（seL4 ケイパビリティ = 偽造不能な権限トークン） |
| IPC | ケイパビリティ起動を伴う同期エンドポイント |
| オブジェクト | CNode、TCB、Endpoint、Notification、VSpace、Frame、Untyped、IRQHandler |
| スケジューラ | MCS（混合臨界度システム）スケジューラ |
| 形式仕様 | Isabelle/HOL の抽象仕様；実行可能仕様へ精緻化；次いで C へ |
| AI フォーカス | なし — 安全性重視の組み込み・サイバーフィジカルシステム |

### 証明チェーン

```
抽象仕様（Isabelle/HOL）
        ↓ （精緻化証明）
実行可能仕様
        ↓ （C 精緻化証明）
C 実装
        ↓ （CompCert による検証コンパイル、部分的）
機械語
```

この階層的証明チェーンは OS 形式検証のゴールドスタンダードです。

## Reikan との類似点

| 観点 | seL4 | Reikan |
|---|---|---|
| **カーネルアーキテクチャ** | マイクロカーネル — 最小限の信頼計算基盤 | 検証済みマイクロカーネル — 最小限の TCB |
| **セキュリティモデル** | ケイパビリティベース；権限は明示的かつ偽造不能 | ケイパビリティ中心；すべての権限が明示的かつ委譲可能 |
| **形式検証** | 正当性・完全性・機密性の機械検証証明 | TCB の完全形式検証が設計目標 |
| **アンビエント権限なし** | ケイパビリティ経由のみのリソースアクセス；バイパス不可 | ケイパビリティ仲介によるアンビエント権限なし |
| **証明ツール** | Isabelle/HOL | Lean 4、Coq、Iris が Reikan の候補 |
| **最小性** | カーネルにはカーネルに必要なことのみ | カーネルにはカーネルに必要なことのみ |
| **IPC 設計** | ケイパビリティを運ぶ IPC メッセージ | ケイパビリティ仲介 IPC |
| **オブジェクトモデル** | 型なしメモリをカーネルオブジェクトに再型付け | 型付きライフサイクルを持つメモリオブジェクトモデル |

## Reikan との相違点

| 観点 | seL4 | Reikan |
|---|---|---|
| **AI フォーカス** | なし — 組み込み安全性重視ワークロード | AI エージェント・LLM パイプライン・モデル推論が第一級の OS 構成要素 |
| **言語** | Isabelle/HOL 証明を伴う C | Reikan-lang — 形式推論と LLM 記述向けに設計された新言語 |
| **異種計算** | GPU/NPU スケジューリングのネイティブサポートなし | GPU・NPU・VRAM・アクセラレータが第一級のスケジュール対象リソース |
| **エージェント抽象化** | エージェント概念なし | 目標・メモリ・ツールアクセス・ライフタイムを持つエージェントがネイティブカーネルオブジェクト |
| **可観測性** | ケイパビリティ仲介に暗黙的；イベントストリームなし | ケイパビリティ付与・モデル呼び出し・ポリシー上書きがすべて監査可能なイベント |
| **開発モデル** | 従来型の学術・産業エンジニアリング | AI 主導開発（LLM がアクティブなコントリビューター） |
| **ターゲットワークロード** | リアルタイム組み込み、航空電子、自動車 | AI 推論・自律エージェント・知識集約型個人計算 |
| **ユーザー空間モデル** | 薄いユーザー空間ライブラリ；ポリシーをユーザーに委ねる | 検証済みカーネルの上に豊富なシステムサービス層 |
| **POSIX** | POSIX 互換レイヤーあり（seL4-POSIX） | 設計上 POSIX レイヤーなし |
| **規模** | 非常に小さなカーネル（約 8,700 行） | カーネル範囲は未定；最小性が目標 |

## 参考にすべき点

### 1. 精緻化証明の方法論
seL4 の証明チェーン（抽象仕様 → 実行可能仕様 → C 実装）は、実際の OS カーネルを形式検証する方法の標準的な例です。Reikan は同様の階層的アプローチを採用すべきです：まず証明支援系で抽象仕様を書き、Reikan-lang に精緻化し、精緻化を機械検証する。

### 2. ケイパビリティオブジェクトモデル
seL4 の小さなカーネルオブジェクト集合（CNode、TCB、Endpoint、VSpace、Frame、Untyped、Notification）と、ケイパビリティの生成・委譲・失効を支配するルールは、Reikan のケイパビリティマネージャの直接的な参照事例です。特に **Untyped メモリ**と再型付け操作は、ランタイム割り当てのあいまいさなしにメモリライフサイクルを明確にモデル化しています。

### 3. CNode — ケイパビリティ空間のナビゲーション
seL4 の CNode モデル（インデックスパスでアドレス指定可能なケイパビリティスロットのカーネル管理ツリー）は、フラットなケイパビリティテーブルに対する有名な代替案です。CNode 型の階層的ケイパビリティ名前空間とフラットテーブルのトレードオフを Reikan の設計で分析する価値があります。

### 4. MCS スケジューラ
seL4 の MCS（混合臨界度システム）拡張は、スケジューリングポリシー（何をいつ実行するか）とスケジューリングドメイン（どのコンポーネントがリアルタイム保証を持つか）を分離します。この関心事の分離は、低レイテンシ AI 推論とバックグラウンドバッチジョブの両方を扱わなければならない Reikan の異種スケジューラに関連します。

### 5. 最小 TCB の規律
seL4 は、カーネルが可能な限り最小限のことをするという規律を強制します：ファイルシステムなし、ネットワークなし、カーネル内デバイスドライバなし。それ以外はすべてユーザー空間にあります。この **TCB 最小性**の原則は、システムサービス層がモデルランタイムやストレージを含むように成長しても、Reikan が守るべき設計制約です。

### 6. 証明ツールチェーンからの教訓
seL4 プロジェクトは大規模な証明エンジニアリングの豊富な経験を生み出しました：大規模な Isabelle 証明スクリプトの管理、カーネル変更時の証明メンテナンス、そして新機能の証明コスト。Reikan の形式検証戦略はこの経験から学ぶべきです — 特に、最初から証明コストを念頭においてカーネルを設計することが重要です。
