# Theseus OS

> English version: [theseus-os.md](theseus-os.md)

---

## 概要

**Theseus OS** は Kevin Boos と Leigh Stoller が率いるバージニア工科大学（後にライス大学）で 2017 年ごろから開発されている研究用 OS です。**イントラリンガル設計**と呼ばれる新しい OS 構造を探求しています：カーネル、ドライバー、アプリケーションコードはすべて同じ言語（Rust）で書かれ、単一アドレス空間にコンパイルされ、ハードウェア特権リングではなくソフトウェア強制の隔離境界のみで区別されます。

Theseus は OS を **セル**と呼ぶ小さな自己完結型の単位に分解します。各セルは明確に定義された状態所有権境界を持つ Rust クレートです。主要な貢献は**状態スピル**の排除です — OS コンポーネントが相互の状態への参照を保持し、隠れた結合を生み出すことで OS の推論・進化・部分更新が困難になる現象です。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| アーキテクチャ | 単一アドレス空間 OS；イントラリンガル；セルベース |
| 主要言語 | Rust — 型システムがセル境界を強制 |
| 隔離メカニズム | ハードウェアリングではなく型レベルの所有権 |
| セルモデル | 各 OS コンポーネント = 所有された状態を持つ隔離された Rust クレート |
| 状態管理 | 状態スピルなし；セルが自身の状態を排他的に所有 |
| ライブ進化 | セルを実行時に交換・更新可能 |
| IPC | 単一アドレス空間内の関数呼び出し（型安全） |
| 形式検証 | なし — Rust の型システムに依存 |
| AI フォーカス | なし — OS 構造研究 |

### 状態スピル問題

従来の OS コンポーネントは互いの内部データ構造への可変参照を保持しています（例：スケジューラがメモリマネージャも保持するタスク構造体へのポインタを保持）。この「状態スピル」により、すべての他のコンポーネントを理解せずに一つのコンポーネントを進化・置換することが不可能になります。Theseus は Rust の所有権ルールを使用して、各セルが自身の状態を排他的に所有することを強制することでこれを排除します。

## Reikan との類似点

| 観点 | Theseus OS | Reikan |
|---|---|---|
| **出自** | Linux のフォークではない；ゼロから設計 | Linux・Windows・macOS のフォークではない |
| **主要言語** | Rust — 構造的特性としての安全性 | Reikan-lang；ブートストラップに Rust を使用 |
| **安全性の哲学** | 型システムが隔離境界を強制 | 言語設計 + 形式証明が不変条件を強制 |
| **分解** | 隔離されたセルの構成としての OS | ケイパビリティ隔離されたコンポーネントの構成としての OS |
| **状態所有権** | Rust による明示的所有権モデル | リソース所有権のための明示的ケイパビリティモデル |
| **マイクロカーネル的** | 最小限の信頼コア；ほとんどのロジックはセルに | 検証済みマイクロカーネル；ほとんどのロジックはシステムサービスに |
| **ライブ進化** | セルを実行時に置換可能 | ポリシー管理されたエージェントライフサイクル管理 |
| **可観測性** | セルレベルのトレースと内省 | すべてのケイパビリティ付与とモデル呼び出しの均一な監査 |

## Reikan との相違点

| 観点 | Theseus OS | Reikan |
|---|---|---|
| **AI フォーカス** | なし — OS 構造研究 | AI エージェント・LLM パイプライン・モデル推論が第一級の OS 構成要素 |
| **形式検証** | なし — 型安全性のみ | TCB の機械検証証明が設計目標 |
| **ケイパビリティセキュリティ** | ケイパビリティベースのセキュリティモデルではない | 完全なケイパビリティセキュリティ — 偽造不能な権限トークン |
| **ハードウェア隔離** | Rust 型システムに依存；ハードウェア特権リングなし | ハードウェア強制特権分離 + 言語安全性 |
| **異種計算** | GPU/NPU スケジューリングなし | GPU・NPU・VRAM・アクセラレータが第一級のスケジュール対象リソース |
| **エージェント抽象化** | エージェント概念なし | 目標・メモリ・ツールアクセス・ライフタイムを持つエージェントがネイティブオブジェクト |
| **アドレス空間モデル** | 単一アドレス空間（SASOS）— パフォーマンス重視 | ハードウェア隔離による別々のアドレス空間 |
| **スコープ** | 研究プロトタイプ | プロダクション向けアーキテクチャ |
| **開発モデル** | 学術研究グループ | AI 主導開発（LLM がアクティブなコントリビューター） |

## 参考にすべき点

### 1. イントラリンガル OS 設計
Theseus の重要な洞察 — OS とそのアプリケーションが一つの言語とコンパイラを共有でき、型システムがハードウェアリングなしで隔離を強制できる — は、注意深く設計された言語で OS インフラを構築することへの強力な論拠です。これは、Reikan-lang をカーネルプリミティブとエージェントレベルのロジックの両方を表現できる言語として設計するという Reikan の決定を支持します。

### 2. 状態スピル防止
**状態スピル**の概念 — 共有可変状態による OS コンポーネント間の隠れた結合 — は Reikan のアーキテクチャの有用なフレーミングです。Reikan は、ケイパビリティマネージャ、スケジューラ、メモリマネージャの内部状態境界が明示的かつ強制されるように設計すべきです：Reikan-lang の所有権型またはケイパビリティ仲介によって。

### 3. セルベース分解
Theseus のセルモデル（各 OS コンポーネントは厳密な状態所有権を持つクレート）は、細粒度の OS 分解の実用的な例です。Reikan では、個々のサブシステム（ケイパビリティマネージャ、スケジューラ、IPC、エージェントランタイム）が形式的に仕様化されたモジュール間インターフェースを持つ隔離されたモジュールとして設計されるべきであることを示唆します。

### 4. ライブ OS 進化
Theseus は再起動せずに実行中の OS コンポーネント（セル）を置換することをサポートします。このライブアップデート特性は Reikan のエージェントライフサイクルモデルにとって興味深いです：エージェントはシステム再起動を必要とせず、ツールライブラリ、モデルウェイト、またはポリシーパラメータを更新できるべきです。

### 5. 単一アドレス空間のトレードオフ
Theseus の SASOS 設計は、ハードウェア隔離を崩すことのパフォーマンス上の利点（TLB シュートダウンなし、OS 内呼び出しのシスコールオーバーヘッドなし）とリスク（型安全性のバグがシステム全体を破損できる）の両方を示しています。Reikan はこれらのトレードオフを明示的に評価すべきです — 特に検証済みカーネルコードとシステムサービスの境界について。
