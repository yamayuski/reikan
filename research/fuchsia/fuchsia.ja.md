# Fuchsia OS

> English version: [fuchsia.md](fuchsia.md)

---

## 概要

**Fuchsia** は Google が開発する汎用オープンソース OS です。Android や Chrome OS とは異なり、Fuchsia は **Zircon** マイクロカーネルをゼロから構築しており、Linux や POSIX、BSD 系統のいかなる流れも汲んでいません。2016 年ごろから開発が始まり、2021 年に Nest Hub スマートディスプレイデバイスに搭載される形で初出荷されました。

Fuchsia の主な設計目標は以下の通りです。

- **ケイパビリティベースのセキュリティモデル**：すべてのリソースアクセスは偽造不能なカーネルハンドルによって仲介される。
- **コンポーネントフレームワーク**：ソフトウェアコンポーネントを、構成・隔離・デプロイの基本単位として扱う。
- 組み込みデバイスからワークステーションまで、複数のハードウェアターゲットをサポートする。
- レガシー POSIX の前提からの決別（オプトインの互換レイヤーは存在する）。

### 主要な技術特性

| 特性 | 詳細 |
|---|---|
| カーネル | Zircon — ケイパビリティベースのマイクロカーネル |
| 主要言語 | C++、Rust |
| IPC | Zircon チャネル（ハンドル転送付きメッセージパッシング） |
| コンポーネントモデル | Component Framework v2 (CF2) — 宣言的マニフェスト、ケイパビリティルーティング |
| インターフェース定義 | FIDL（Fuchsia インターフェース定義言語） |
| ビルドシステム | GN + Ninja |
| セキュリティモデル | ケイパビリティハンドル；アンビエント権限なし；パッケージのシール |
| 形式検証 | 部分的（一部カーネル不変条件；カーネル全体の機械検証証明はなし） |
| AI フォーカス | なし — 汎用ワークロード向け |

## Reikan との類似点

| 観点 | Fuchsia | Reikan |
|---|---|---|
| **カーネルの系譜** | 非POSIX・非Linux のゼロから設計 | 非POSIX・非Linux のゼロから設計 |
| **セキュリティモデル** | ケイパビリティベース；アンビエント権限なし | ケイパビリティ中心；アンビエント権限なし |
| **アーキテクチャ** | マイクロカーネル（Zircon）＋上位システムサービス | 検証済みマイクロカーネル＋システムサービス層 |
| **コンポーネントモデル** | コンポーネントをランタイムの基本単位とする | エージェントを OS の第一級市民とする |
| **IPC** | ケイパビリティハンドル付きチャネルベース | ケイパビリティ仲介 IPC |
| **レガシーからの決別** | 設計上 POSIX 義務なし | POSIX・Win32・ELF の義務なし |
| **Rust の採用** | C++ と並ぶ第一級言語として Rust を採用 | Reikan-lang は LLM フレンドリー・安全設計；ブートストラップに Rust を使う可能性あり |
| **インターフェース定義** | FIDL — 型付き・バージョン管理 IDL | Reikan 独自のインターフェース層が必要 |

## Reikan との相違点

| 観点 | Fuchsia | Reikan |
|---|---|---|
| **AI フォーカス** | 汎用；AI 固有の抽象化なし | AI エージェント・モデル推論・LLM パイプラインが第一級の OS 構成要素 |
| **形式検証** | 部分的；カーネル全体の機械検証証明はなし | TCB の完全形式検証が設計目標 |
| **主要言語** | C++ と Rust（既存言語） | Reikan-lang — LLM 記述・形式推論向けに設計された新言語 |
| **異種計算** | GPU/NPU スケジューリングのネイティブサポートは限定的 | GPU・NPU・VRAM・アクセラレータが第一級のスケジュール対象リソース |
| **可観測性** | コンポーネントイベントログ；統一監査ではない | ケイパビリティ付与・モデル呼び出し・ポリシー上書きがすべて監査可能なイベント |
| **ターゲットワークロード** | 組み込みからデスクトップ；汎用アプリケーション | AI 推論・自律エージェント・知識集約型計算 |
| **開発モデル** | 従来型エンジニアリングチーム | AI 主導開発（LLM がアクティブなコントリビューター） |
| **エージェント抽象化** | エージェント概念なし | 目標・メモリ・ツールアクセス・ライフタイムを持つエージェントがネイティブカーネルオブジェクト |

## 参考にすべき点

### 1. ケイパビリティハンドルモデル（Zircon）
Zircon のカーネルオブジェクトハンドルと明示的な権限は、ケイパビリティセキュリティの最も明確なプロダクション実装の一つです。IPC チャネルを通じたハンドル転送のセマンティクス、権限のシールの概念、そして**アンビエント権限なし**の方針は、Reikan のケイパビリティテーブルと委譲規則を設計する際に参照すべきです。

### 2. FIDL — Fuchsia インターフェース定義言語
FIDL は、コンポーネント間の型付き・バージョン管理・トランスポート非依存のインターフェース契約を提供します。複数言語のバインディングを生成し、ワイヤー互換性ルールを強制します。Reikan はエージェント間・コンポーネント間の契約に相当する IDL 層が必要です；FIDL の進化モデル（API 互換性と ABI 互換性）は参考になります。

### 3. コンポーネントフレームワーク v2（CF2）
CF2 はコンポーネントの**トポロジー**（誰が誰に何を提供できるか）とコンポーネントの**インスタンス化**を分離します。ケイパビリティルーティングはコードにハードコードされず、コンポーネントマニフェストに宣言されます。このポリシーとメカニズムの明確な分離は、Reikan のケイパビリティマネージャ設計に直接関連します。

### 4. シールドパッケージモデル
Fuchsia のパッケージはビルド時にコンテンツアドレス指定でシールされます。ケイパビリティルーティングと組み合わせることで、任意のコンポーネントがアクセスできるものを制約します。Reikan のエージェントデプロイに同様のシールドアーティファクトモデルが適切かもしれません。

### 5. ドライバーフレームワーク（DFv2）
Fuchsia の最新ドライバーフレームワークは、ドライバーを特権的なカーネルモジュールではなく、宣言されたケイパビリティを持つコンポーネントとしてモデル化します。これは Reikan のハードウェア抽象化層設計の有用な参照事例です。
